# Auth + Flow Authoring Improvements (Brainstorm)

## Problem Summary

Coworkers are struggling with the current setup path:

1. Build a collection in Postman
2. Configure auth in Postman (including browser-based OAuth/implicit flows)
3. Execute in Postman/Newman
4. Export collection + environment JSON
5. Run those files through Bondtrace recorder

This is especially painful for OAuth scenarios where auth can require browser redirects and hand-managed tokens.

## Goals

- Reduce dependence on Postman as the authoring surface
- Make auth setup explicit and guided
- Preserve Bondtrace strengths: deterministic chains, templating, variable exports
- Enable a format that AI assistants can generate/review/edit safely

## Option A: Keep Postman, Add “Auth Helpers” Around It

### What to add

- **Preflight validator CLI**
  - Checks collection/environment for known failure modes:
    - Missing oauth token variables
    - Missing base URLs
    - Broken `{{var}}` references
- **Auth bootstrap commands**
  - `bondtrace auth oauth2-client-credentials --env env.json --profile staging`
  - Fetches token and writes env values in expected schema
- **Recorder diagnostics mode**
  - On 401/403, prints likely cause and suggested fix

### Pros

- Lowest migration risk
- Leverages teams’ existing Postman assets

### Cons

- Core complexity remains in Postman
- Browser-based auth still awkward

## Option B: Introduce a Bondtrace Native Flow Spec (Recommended)

Create a human/AI friendly format (JSON preferred) for request chains, variables, exports, and auth profiles.

### Why this helps

- Removes tool-hopping (Postman -> Newman -> Bondtrace)
- Can be generated by AI from OpenAPI + user inputs
- Auth can be declarative and handled centrally

## Proposed Spec Shape (Draft)

```yaml
version: 1
name: account-onboarding-demo

inputs:
  baseUrl:
    type: string
    required: true
  clientId:
    type: secret
    required: true
  clientSecret:
    type: secret
    required: true
  username:
    type: secret
    required: false
  password:
    type: secret
    required: false

auth:
  profiles:
    serviceClient:
      type: oauth2-client-credentials
      tokenUrl: "{{baseUrl}}/oauth/token"
      clientId: "{{clientId}}"
      clientSecret: "{{clientSecret}}"
      scopes: [api.read, api.write]
      export:
        accessToken: auth.serviceClient.access_token

    browserUser:
      type: oauth2-implicit
      authorizeUrl: "{{baseUrl}}/authorize"
      clientId: "{{clientId}}"
      redirectUri: "http://localhost:7777/callback"
      scopes: [openid, profile]
      export:
        accessToken: auth.browserUser.access_token

steps:
  - id: createCustomer
    request:
      method: POST
      url: "{{baseUrl}}/customers"
      authProfile: serviceClient
      headers:
        Content-Type: application/json
      body:
        firstName: Jane
        lastName: Doe
    export:
      customerId: response.body.id

  - id: getCustomer
    request:
      method: GET
      url: "{{baseUrl}}/customers/{{customerId}}"
      authProfile: browserUser
```

## Auth Handling Strategy

### OAuth2 Client Credentials

- Fully non-interactive
- First-class resolver in recorder runtime:
  - Acquire token
  - Cache by profile + scope + audience
  - No refresh in v1 (re-acquire next run)

### OAuth2 Implicit / Browser Login

- Add **Bondtrace Auth Broker** (local callback mini-server)
  - Opens browser for login
  - Captures redirect hash/token
  - Stores token in secure local cache for run
- Support both Implicit and Authorization Code + PKCE modes for browser-based auth

### Credential UX

- `bondtrace run flow.v1.json -n dev --env-file .env.dev`
- Support `.bondtrace.secrets.json` (gitignored)
- Optional OS keychain integration later

## AI-Native Workflow

- Runner should resolve required keys up front from selected environment + auth profiles before any request is sent.


### Command concepts

- `bondtrace init-flow --from-openapi api.yaml`
  - Generates draft flow with operation IDs and sample bodies
- `bondtrace doctor flow.v1.json -n local --env-file .env.local`
  - Emits warnings (single warning type) for missing required keys, variable graph issues, auth profile mismatches, and unbound templates
- `bondtrace run flow.v1.json -n sandbox --env-file .env.sandbox --record tape.json`
  - Executes chain and emits Bondtrace tape directly

### AI interaction model

- Include non-request timeline steps for presenter pacing (wait N seconds, wait for manual continue).

- Keep spec deterministic and explicit
- Provide machine-checkable diagnostics (line/field-targeted errors)
- Emit “fix suggestions” that an AI can apply safely

## Backward Compatibility

- Keep existing `record <collection.json>` path
- Add converter:
  - `bondtrace convert postman collection.json -e env.json > flow.v1.json`
- Teams can migrate gradually

## Suggested Rollout

### Phase 1: Fast Relief

- Preflight validator for Postman exports
- Better auth-related error messages

### Phase 2: New Native Spec MVP

- JSON spec parser
- client credentials auth profile
- step export/template runtime
- run -> tape generation

### Phase 3: Interactive Auth + AI Tooling

- local auth broker for implicit/code+pkce
- openapi scaffolder
- doctor/autofix outputs for AI loops

## Risks & Mitigations

- **Risk:** New spec increases maintenance burden
  - **Mitigation:** Keep v1 minimal and map closely to current runtime model
- **Risk:** OAuth edge cases across providers
  - **Mitigation:** auth provider plugin interface + tested reference providers
- **Risk:** Security of secrets/tokens
  - **Mitigation:** secure local storage, redaction in logs, explicit secret types

## Practical Recommendation

Pursue **Option B** (native flow spec) while shipping a small subset of Option A immediately.

- Short term: reduce pain with validators + diagnostics
- Medium term: eliminate Postman bottleneck and unlock AI-authored flows
- Long term: treat Bondtrace as the source of truth for demo flow execution and playback recording
